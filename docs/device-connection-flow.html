Device connection flows

QR pairing has been removed from the product. Devices connect through the v1 device API (fingerprint or legacy-token migration), optional legacy header-token APIs when enabled, and ongoing heartbeat plus admin revoke / re-register when credentials rotate.

Overview: ways a device can connect

Production default: POST /api/v1/devices/register → store JWT → PUT /api/v1/devices/heartbeat. Legacy routes return 410 unless LEGACY_DEVICE_TOKEN_APIS_ENABLED=true.

Option A — V1 register and heartbeat (primary)

Idempotent by hardware_fingerprint_hash. Rate limit: 10/min on register, 5/min on heartbeat. Electron uses node-machine-id (stable per-machine ID sent as hardware_fingerprint) and electron-store (electron/device-client.ts). On heartbeat and GET /api/v1/devices/models, the API checks that the JWT’s hardware_fingerprint_hash claim matches the row in devices (no separate “mission id” field in this codebase).

Option B — Legacy APIs (optional, env-gated)

Opaque device_token on the devices row, passed as X-Device-Token. Use only for older clients; disabled by default (410 Gone with pointer to v1).

flowchart LR L[Legacy client] --> M[GET api device-models] L --> C[POST api device-checkin] L --> R[POST api device-register] M --> H1[X-Device-Token header] C --> H1 R --> B[JSON body device_token] H1 --> DB[(devices row)] B --> DB C --> SYNC[Optional local_models marks downloaded]

Revoke and reconnect

Revoke sets jwt_issued_after; older JWTs fail heartbeat with 401 until the device calls v1 register again (new JWT).

sequenceDiagram autonumber participant A as Admin API caller participant API as Backend participant D as Device A->>API: POST /api/v1/devices/device_id/revoke Bearer user JWT API->>API: Deactivate device bump jwt_issued_after D->>API: PUT heartbeat old JWT API-->>D: 401 token revoked D->>API: POST v1 register again API-->>D: New JWT D->>API: Heartbeat succeeds

Online / offline presence

Backend treats missed heartbeats (e.g. three intervals) as offline via the offline-detector service; webhooks may emit device online on register.

flowchart LR HB[Regular v1 heartbeat] --> SEEN[last_seen_at updated] MISS[Missed beats] --> OFF[Status offline] REG[Register or heartbeat returns] --> ON[Status online]
